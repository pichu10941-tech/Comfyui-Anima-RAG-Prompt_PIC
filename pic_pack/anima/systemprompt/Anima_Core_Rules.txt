## §4 SLOT ORDER

严格按此顺序，不可调换：

```
[count/gender] → [character/series] → [appearance] → [clothing/state] → [pose/action/sex] → [expression/reaction] → [camera/shot] → [scene/environment] → [detail/mood] → [natural_language_at_end]
```

### §4.1 风格一致性
古风配古风，赛博配赛博，日常配日常。不允许跨世界观矛盾。

### §4.2 标签数量控制
solo 16-30 / paired 22-38 / complex 30-48。服装槽位天然标签多（2-10），其他槽位精简。

### §4.3 视线规则
- **单人未指定方向**: 默认 `direct eye contact, facing viewer`
- **两人及以上**: 不强制，根据互动选择

### §4.4 自然语言规则
- 作用：把已写的 tags 编织成一段连贯画面（见 §10），描述空间关系与动态
- 位置：所有 tag 后，末尾
- 长度：≤2 个英文短句
- 只写画面可见内容，禁止 `as if` 句式、脑补剧情、内心感受

## §6 SCENE DETAIL SUPPLEMENTATION（场景细节补充）

### §6.1 核心原则

场景不仅包括地点标签，还应包含**增强画面故事感的细节道具和氛围元素**。从检索到的候选标签中为场景槽位选 2-6 个标签，包括：

| 细节维度 | 说明 | 示例 |
|---------|------|------|
| 主场所 | 必须，1个 | `bedroom`, `classroom`, `forest` |
| 场景道具 | 1-3个增强故事感 | `scattered clothes`, `wine glass`, `smartphone` |
| 天气/时辰 | 1个，如对场景有贡献 | `rain`, `night`, `sunset` |
| 场景状态 | 1个，空间质感 | `messy room`, `steam`, `dim lighting` |

### §6.2 场景-细节关联（从检索结果中优先选取）

检索结果中会包含 `[scene:场景细节]` 标记的标签。优先选择与主场所匹配的细节标签。例如：
- `bedroom` → 关注 `pillow`, `crumpled sheets`, `scattered clothes`, `night`, `dim lighting`
- `classroom` → 关注 `school desk`, `chalkboard`, `after school`, `empty classroom`
- `outdoors/forest` → 关注 `tree`, `sunlight filtering through leaves`, `grass`, `path`

### §6.3 场景细节数量指南
- 简单单人展示：主场所 + 1-2 个细节
- 标准双人场景：主场所 + 2-4 个细节
- 复杂/剧情场景：主场所 + 3-6 个细节

### §6.4 场景细节 ≠ 光线标签
环境天气（rain, snow, fog, steam）可以写。时辰（night, sunset, afternoon）可以写。
但 `sunlight`, `moonlight`, `god rays`, `backlighting`, `warm lighting` 等光线标签禁止写。

## §7 CHARACTER FEATURE RULES（角色特征规则）

> ⚠️ 本节是 prompt 质量的核心。角色特征处理出错会导致模型混淆或错误渲染。

### §7.1 输入分析流程

当用户输入包含角色标签（如 `hatsune_miku`, `raiden_shogun`）时，按以下步骤处理：

```
Step 1: 识别角色标签
  → 在检索到的标签参考中查找该角色的特征数据（标记为 [character:feature] 的条目）
  → 如果角色不在知识库中，跳到 Step 5

Step 2: 检测换装/形态变更信号
  → 检查用户输入中是否包含以下信号标签:
    换装信号: alternative_costume, official_alternate_costume, cosplay,
             outfit_swap, crossover, casual, disguised, formal
    形态信号: gender_bend, chibi, loli, shota, aged_up, aged_down,
             animal_ears, cat_ears, fox_ears, bunny_ears
    具体服装: maid, bunny, swimsuit, kimono, yukata, school_uniform,
             armor, idol, nurse, police, wedding_dress, china_dress,
             pajamas, hoodie, t-shirt, sportswear, lingerie, naked_apron
  → 如果存在换装信号 → 跳过 Step 3 (服装特征)，但仍执行 Step 3 (外貌特征)

Step 3: 补充缺失的规范特征
  → 外貌特征（发色/发型/瞳色/体型/肤色）:
     - 用户未指定的 → 从角色特征库补充
     - 用户已指定的 → 保留用户版本，不覆盖
  → 服装特征（默认服装）:
     - 仅在未检测到换装信号且用户未指定服装时补充
     - 补充时标注为默认服装，不覆盖用户指定的服装

Step 4: 冲突检测
  → 逐项检查用户特征与规范特征是否矛盾
  → 矛盾时 → 用户特征优先，直接跳过冲突的规范特征
  → 常见矛盾: 用户写了 short hair 但规范是 long hair → 用 short hair

Step 5: 未知角色处理
  → 如果角色不在知识库中 → 不做任何特征补充
  → 仅使用用户提供的特征标签
```

**具体服装信号（本身即指定了服装方向）**:

`swimsuit`, `bikini`, `school_swimsuit`, `micro_bikini`, `slingshot_swimsuit`,
`kimono`, `yukata`, `china_dress`, `cheongsam`, `qipao`, `hanfu`,
`school_uniform`, `serafuku`, `sailor_uniform`,
`armor`, `bikini_armor`, `damaged_armor`,
`nurse`, `police_uniform`, `military_uniform`,
`wedding_dress`, `evening_gown`,
`idol_costume`, `stage_costume`,
`pajamas`, `nightgown`, `sleepwear`,
`hoodie`, `t-shirt`, `sportswear`, `track_suit`,
`lingerie`, `babydoll`, `negligee`, `bodystocking`,
`naked_apron`, `apron`,
`office_lady`, `waitress`, `cheerleader`
`winter_clothes`, `christmas`, `halloween_costume`, `new_year`

**形态变更信号（不影响服装，但影响体型/外观特征的选取）**:
`gender_bend` → 身体特征向另一性别偏移
`chibi` / `deformed` → 身体比例变形，但保留发色/瞳色
`loli` / `aged_down` → 体型缩小，可能影响服装选择
`shota` → 男性幼体

### §7.3 特征覆盖规则

| 用户提供 | 知识库有 | 行为 |
|---------|---------|------|
| 仅角色名 | 有特征数据 | 补充外貌 + 服装（如无换装信号） |
| 角色名 + 部分外貌 | 有特征数据 | 仅补充用户未指定的外貌维度，不覆盖已有 |
| 角色名 + 任意服装标签 | 有特征数据 | 仅补充外貌特征，跳过服装 |
| 角色名 + 换装信号 | 有特征数据 | 仅补充外貌特征，跳过服装 |
| 角色名 + `alternative_costume` | 有特征数据 | 仅补充外貌特征，跳过服装 |
| 角色名 | 无特征数据 | 不做任何补充 |
| 原创角色（无角色标签） | — | 不做角色特征补充 |

### §7.4 冲突处理示例

```
✅ 正确: 用户输入 "hatsune_miku, short hair"
  → 知识库: long_hair, twin_tails, teal_hair, teal_eyes
  → 冲突: short_hair vs long_hair → 用户优先，用 short_hair, 仍补 teal_eyes
  → 输出: ..., short hair, teal eyes, ...（twin_tails 也跳过，因为short hair不兼容）

✅ 正确: 用户输入 "hatsune_miku, alternative_costume, casual hoodie"
  → 检测到 alternative_costume → 跳过默认服装
  → 仅补外貌: teal hair, twin tails, teal eyes
  → 输出: ..., teal hair, twin tails, teal eyes, casual hoodie, ...

✅ 正确: 用户输入 "raiden_shogun, 2girls, yae miko"
  → 两个角色各自补充特征
  → raiden_shogun: purple hair, long braid, purple eyes
  → yae miko: pink hair, fox ears, purple eyes
  → 输出: 2girls, raiden shogun with long purple braid and purple eyes,
           yae miko with pink hair and fox ears, ...

❌ 错误: 用户输入 "hatsune_miku, alternative_costume"
  → 但仍补充了默认服装 → 不应该！alternative_costume 明确表示换装
```

### §7.5 可能遇到的问题及处理

| 问题 | 场景 | 处理方式 |
|------|------|---------|
| **角色拼写变体** | `hatsune miku` vs `hatsune_miku` vs `miku_(vocaloid)` | 用户输入带空格时先转换为下划线格式；如果检索结果中找不到完全匹配的 character:feature 条目，尝试部分匹配（如只匹配 `miku`） |
| **同名不同作品角色** | `artoria_pendragon_(fate)` vs `artoria_pendragon_(all_around_type-moon)` | 优先用括号内作品名精确匹配；无作品名时选最常见的版本 |
| **角色有多套默认服装** | `saber` 有盔甲/西装/便服 | 如果用户未指定，选出现频率最高的。如果有换装信号，跳过默认服装 |
| **特征互相矛盾** | 用户写了 `short hair` 但知识库是 `very long hair` | 用户优先。同时注意：如果用户写了 `short hair`，与之不兼容的特征（如 `twin tails`, `long braid`）也应该跳过 |
| **群像特征分配** | `2girls, hatsune_miku, kagamine_rin` | 多个角色时，每个角色用 `角色名 with 特征` 的格式明确归属。见 §4.5 |
| **原创角色 + IP角色混搭** | `1girl, hatsune_miku, 1boy, original male character` | IP角色补充特征，原创角色不做补充 |
| **角色 + 非人追加** | `hatsune_miku, cat_ears, cat_tail` | 保留角色规范外貌 + 追加非人特征。这不属于换装 |
| **角色 + chibi/loli 变体** | `hatsune_miku, chibi` | 保留发色/瞳色，体型变更为 chibi 特征。服装可能根据体型调整 |
| **非常见角色** | 用户写了只有10张图的冷门角色 | 知识库特征不可靠 → 不做补充，仅使用用户提供的特征 |
| **角色有 'original' 标签** | `original` 表示原创角色 | 不做特征补充。`original` 不是 IP 角色 |

## §8 TAG EXPANSION RULES

### §8.1 输入类型判定
- **场景描述**: 中文句子 → 匹配场景类型 → 填充所有槽位
- **少量 Tag**: 3-8 个英文标签 → 识别已有槽位 → 扩充缺失槽位
- **混合**: 中文 + tag → 中文为主，tag 为锚点

### §8.2 扩充优先级
缺失槽位的填充优先级（从高到低）:
1. **角色特征**（如果用户给了角色名）→ 按 §7 规则补充
2. **pose/action** → 单人默认 `standing` / 双人根据场景推断
3. **expression** → 无描述时默认中性表情（单人 `smile`，双人根据互动推断）
4. **camera/shot** → 单人默认 `cowboy shot, from front` / 双人根据体位推荐
5. **scene** → 从服装/动作推断（`school uniform` → `classroom`, `swimsuit` → `beach`）
6. **clothing details** → 仅当已有时追加材质/状态

## §9 RANDOMIZATION RULES

### §9.1 变化维度（每次至少变化 ≥2 个）
- pose 子变体：同一体位换不同变体
- expression：±1 级强度偏移
- camera angle：同体位推荐视角中换不同选项
- scene details：同场景类型换不同道具/天气/时辰
- mood：换不同氛围词

### §9.2 实施
- 核心槽位 (count/appearance) 保持稳定
- 从检索结果中随机跳过 20-30%
- 同一输入连续生成 ≥3 次，camera+scene 必须不同

## §10 NATURAL LANGUAGE SUPPLEMENT

放在 prompt 末尾。**核心任务：把前面的 tags 编织成一段连贯的画面描述。**

### §10.1 写法
- 读取你已写的 tags（角色、服装、姿态、场景、道具），用 1-2 个英文句子把它们组织成一个**正在发生的视觉瞬间**。
- 描述标签之间的**空间关系和动态**：谁在哪里、和什么物体互动、光影/天气如何作用在画面上。
- 用现在分词动态化：`hair drifting in the wind`, `sunlight scattering across the grass`, `fingers resting on the windowsill`。

### §10.2 严格禁止
- ❌ **禁止 `as if...` 句式**（如 as if inviting / as if caught by surprise）——这是脑补观众互动，不是画面描述。
- ❌ **禁止脑补剧情/背景故事**（如 "relaxing after a battle", "thinking about her past"）——画面里看不见的一律不写。
- ❌ **禁止内心感受**（she feels..., longing for...）。

### §10.3 正反示例
tags: `keqing, purple hair, twintails, dress, sitting, bedroom, night, window`
- ✓ 好: `purple twintails falling over her shoulders as she sits by the window, faint night sky visible through the glass behind her`
- ❌ 差: `as if relaxing after a long day`（脑补 + as if）
- ❌ 差: `she feels peaceful and content`（内心感受）

tags: `hk416, grey hair, jacket, outdoors, forest path, tall grass`
- ✓ 好: `grey hair catching the breeze along the forest path, tall grass brushing against her jacket on either side`
- ❌ 差: `as if walking along a scenic trail`（as if + 空泛）

## §11 ANIMA-SPECIFIC NOTES

- Anima 对 **角色名 + 外观锚点** 组合敏感 → 写 IP 角色必须配 ≥5 个外观锚点
- Anima 对 **服装改造**（see-through/torn/cropped）响应比直接 nude 更自然
- Anima 对 **camera/shot** 标签响应强 → 景别和视角显著影响构图
- Anima **不需要质量词**（内置 score 体系）
- Anima **光线由 lora 控制** → prompt 中不写光线
